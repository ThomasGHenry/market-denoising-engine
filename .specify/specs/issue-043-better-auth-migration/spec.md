# Spec: Better Auth Migration

**Issue:** #43
**ADR:** docs/adr/0108-migrate-auth-js-to-better-auth.md
**Status:** clarified

---

## 1. Purpose

Replace Auth.js v5 (`next-auth@beta`, `@auth/prisma-adapter`) with Better Auth
(`better-auth`) to eliminate five structural defects in the current auth
implementation: the mandatory two-file edge split, the unpooled-connection
runtime requirement, the `trustHost: true` production exposure, broken GitHub
OAuth, and an E2E helper that cannot authenticate.

---

## 2. Acceptance Criteria

- [ ] Magic link flow: user receives email, clicks link, lands on protected page
- [ ] GitHub OAuth flow: user clicks Sign in with GitHub, authorises, lands on protected page
      (gated on `AUTH_GITHUB_ID` env var presence)
- [ ] Middleware protects all non-public routes; unauthenticated request â†’ redirect to `/login`
- [ ] E2E smoke test `loginWithMagicLink` passes in CI
- [ ] `auth.config.ts` deleted; zero references to it remain in the codebase
- [ ] `DATABASE_URL_UNPOOLED` removed from runtime auth (still acceptable in infra migration scripts)

---

## 3. Functional Requirements

### FR-001 â€” Single auth entry point

A single `apps/web/src/lib/auth.ts` exports the Better Auth instance and
all necessary auth helpers. No `auth.config.ts` exists.

### FR-002 â€” Magic link email

User submits their email address on `/login`. Better Auth sends a magic
link email via Resend. User clicks the link and is authenticated; browser
lands on `/`.

### FR-003 â€” GitHub OAuth (presence-gated)

When `AUTH_GITHUB_ID` is defined in the environment, a "Sign in with
GitHub" button is rendered on `/login`. Clicking it initiates the GitHub
OAuth flow. On success the user is authenticated and lands on `/`.

### FR-004 â€” Route protection

All routes other than `/login` and `/api/auth/*` require an authenticated
session. An unauthenticated request redirects to `/login`.

### FR-005 â€” E2E testability

The `loginWithMagicLink(page, email)` helper in `apps/web-e2e/src/auth.ts`
must authenticate programmatically without a real email client.

Better Auth hashes the raw token before storing it in the `verification` table's
`identifier` column. The raw token is available in the `sendMagicLink` callback
as `{ token, url }`. The `url` is the full verification URL containing the raw token.

The `sendMagicLink` callback in `auth.ts` is the interception point. Implementation:

1. When `process.env.NODE_ENV !== 'production'`, write the raw `url` to
   an additional Prisma `Verification` row with a known sentinel identifier
   (`__e2e_magic_link__<email>`). Better Auth does not prohibit additional rows.
2. E2E helper: POST to `/api/auth/sign-in/magic-link`, query the DB for the
   sentinel row, extract the URL from its `value` field, navigate to it.

This adds no new Prisma models and works in both local and Vercel preview environments
since both share the same Neon database.

E2E helper flow:
1. POST `{ email, callbackURL: '/' }` to `<baseUrl>/api/auth/sign-in/magic-link`
2. `prisma.verification.findFirst({ where: { identifier: '__e2e__' + email } })`
   â†’ value field contains the full URL
3. `page.goto(url)` â†’ authenticated session established
4. Verify `page.url()` is `<baseUrl>/`

### FR-006 â€” Schema replacement

The Auth.js `User`, `Account`, `Session`, `VerificationToken` models are
replaced by the Better Auth schema (`user`, `session`, `account`,
`verification`). A Prisma migration is generated and applied.
